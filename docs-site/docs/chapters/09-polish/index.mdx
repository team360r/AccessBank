## Section 2: Platform Differences

Each platform's screen reader has its own quirks. Something that works perfectly on Android may behave differently on iOS. Both platforms need explicit testing before shipping.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

**TalkBack (Android)**
- Uses "explore by touch" by default — swiping to navigate is the primary modality
- `liveRegion: true` is well supported and reliable
- Long-swipe gestures can be configured by the user in TalkBack settings
- Toggle: press both volume buttons simultaneously, or triple-press the side button on some devices

**VoiceOver (iOS)**
- Toggle: triple-press the side button (iPhone X and later) or home button
- `hint` text is read after a brief pause following the label, or on demand via the Rotor gesture
- Announcements from `SemanticsService` can sometimes be delayed by a few hundred milliseconds — test on device, not simulator
- The Rotor (rotate two fingers on screen) provides quick navigation by headings, links, and custom actions

</div>
</div>

<div className="ab-callout ab-callout--amber">
  <div className="ab-callout__header">Why this matters</div>
  <p>An issue invisible on Android may be a showstopper on iOS, and vice versa. TalkBack's "explore by touch" and VoiceOver's swipe navigation feel similar but have different timing and gesture sensitivity. Always test on both physical devices — simulators are insufficient for audio-based accessibility testing.</p>
</div>

## Section 3: Custom Semantics Actions

Beyond tap and long-press, `CustomSemanticsAction` lets you expose custom actions that screen readers can trigger via their action menus — VoiceOver's Rotor, TalkBack's local context menu.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

This is perfect for features like "Copy reference number" or "Report an issue" — actions that have no obvious button in the visual layout but that screen reader users should still be able to reach without hunting through a context menu.

  </div>
</div>

<Tabs>
  <TabItem value="before" label="Before — tap and long-press only" default>

```dart title="lib/screens/transactions/widgets/transaction_tile.dart"
// Only tap and long-press available — hidden features for screen reader users
GestureDetector(
  onTap: () => _openDetails(transaction),
  onLongPress: () => _showMenu(transaction),
  child: TransactionTileContent(transaction: transaction),
)
```

  </TabItem>
  <TabItem value="after" label="After — custom actions in screen reader menu">

```dart title="lib/screens/transactions/widgets/transaction_tile.dart"
// Custom actions appear in TalkBack's local context menu
// and VoiceOver's Rotor actions
Semantics(
  label: '${transaction.description}, '
         '${transaction.formattedAmount}',
  customSemanticsActions: {
    const CustomSemanticsAction(label: 'Copy reference number'):
        () => _copyReference(transaction),
    const CustomSemanticsAction(label: 'Report an issue'):
        () => _reportIssue(transaction),
  },
  child: GestureDetector(
    onTap: () => _openDetails(transaction),
    child: TransactionTileContent(transaction: transaction),
  ),
)
```

  </TabItem>
</Tabs>

<div className="ab-callout ab-callout--blue">
  <div className="ab-callout__header">💡 Try it yourself</div>
  <p>With TalkBack active, navigate to a transaction tile and open the local context menu (swipe up then right, or use the TalkBack menu button). You should see "Copy reference number" and "Report an issue" listed as actions alongside the default Tap and Long-press options.</p>
</div>

## Section 4: Keeping the Semantics Tree Lean

A large semantics tree has a performance cost — Flutter must diff and synchronise it on every frame that changes. Keep the tree lean by removing unnecessary nesting.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

Guidelines for a lean semantics tree:

- Use `ExcludeSemantics` on decorative subtrees (icons, separators, background images)
- Use `MergeSemantics` to collapse related nodes into a single announcement
- Avoid wrapping every `Text` in its own `Semantics` widget when `MergeSemantics` on the parent achieves the same result
- Profile with `showSemanticsDebugger: true` in `MaterialApp` — if you see more nodes than you expect, look for unnecessary wrappers

</div>
</div>

<Tabs>
  <TabItem value="before" label="Before — 5 separate semantics nodes" default>

```dart title="lib/screens/account_overview/widgets/account_card.dart"
// Every child widget generates its own semantics node
Semantics(
  child: Row(
    children: [
      Semantics(child: Icon(Icons.account_balance)),
      Semantics(child: Text(account.name)),
      Semantics(child: Text(account.formattedBalance)),
      Semantics(child: Icon(Icons.chevron_right)),
    ],
  ),
)
```

  </TabItem>
  <TabItem value="after" label="After — 1 merged semantics node">

```dart title="lib/screens/account_overview/widgets/account_card.dart"
// One clean node per card — decorative icons excluded
MergeSemantics(
  child: Row(
    children: [
      ExcludeSemantics(child: Icon(Icons.account_balance)),
      Text(account.name),
      Text(account.formattedBalance),
      ExcludeSemantics(child: Icon(Icons.chevron_right)),
    ],
  ),
)
```

  </TabItem>
</Tabs>

<div className="ab-callout ab-callout--amber">
  <div className="ab-callout__header">Why this matters</div>
  <p>On the Account Overview with three account cards, the difference is between 15+ semantics nodes versus 3. Every unnecessary node adds overhead to the semantics diff on each frame, and adds unnecessary swipe stops for screen reader users. Lean trees are both faster and more pleasant to use.</p>
</div>

## Section 5: Resources for the Journey

Accessibility is not a destination — it is a continuous practice.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

**Flutter-specific**
- [Flutter Accessibility documentation](https://docs.flutter.dev/accessibility) — comprehensive API coverage and best practices
- [Flutter DevTools](https://docs.flutter.dev/tools/devtools) — inspector, semantics overlay, and performance tooling

**Standards**
- [WCAG 2.1](https://www.w3.org/TR/WCAG21/) — the web/app accessibility standard; Level AA is the legal baseline in most jurisdictions
- [W3C Mobile Accessibility Guidelines](https://www.w3.org/TR/mobile-accessibility-mapping/) — mobile-specific patterns and considerations

**Platform testing tools**
- [Google Accessibility Scanner](https://play.google.com/store/apps/details?id=com.google.android.apps.accessibility.auditor) — Android automated audit tool
- [Xcode Accessibility Inspector](https://developer.apple.com/videos/play/wwdc2019/257/) — iOS and macOS inspection and testing
- [Colour Contrast Analyser](https://www.tpgi.com/color-contrast-checker/) — desktop app for checking contrast ratios

</div>
</div>
