# See Clearly

> *"My grandma could actually use this now."*

**Estimated time:** ~20 minutes | **Focus:** Account Overview + Theme | **Branch:** `chapter-4-visual`

---

Visual accessibility goes far beyond screen readers. In this chapter you'll tackle the problems that affect users who *can* see the screen — just not as clearly as you might assume. That includes older adults, people with low vision, users in bright sunlight, and anyone who's increased their device font size.

There are more of these users than you might think. One in three people over 65 have some form of vision impairment. And everyone — regardless of vision — benefits from text they can actually read.

---

## Section 1: Contrast Matters

The original AccessBank colour scheme uses `AppColors.inaccessible.lightBlueOnWhite` (`#90CAF9`) for account names and balances on white cards. That's a contrast ratio of roughly **2.0:1** against white. WCAG 2.1 requires **4.5:1** for normal-sized body text.

That's not a near-miss. It's less than half the required contrast.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

WCAG 2.1 Success Criterion 1.4.3 sets these minimum contrast ratios:

- **4.5:1** for normal text (below 18pt regular or 14pt bold)
- **3:1** for large text (18pt regular or 14pt bold and above)
- **3:1** for user interface components and graphical objects

These aren't arbitrary numbers. They represent the threshold at which most people with moderate vision loss — including many older adults — can still read text comfortably.

  </div>
</div>

<div className="ab-step">
  <div className="ab-step__badge">2</div>
  <div className="ab-step__content">

The fix in AccessBank is to switch from the inaccessible palette to `AppColors.primary` and `AppColors.textPrimary` — deep blue and near-black respectively, both with contrast ratios well above 4.5:1 against the white card background.

  </div>
</div>

<Tabs>
  <TabItem value="before" label="Before — fails WCAG 1.4.3" default>

```dart title="lib/screens/account_overview/widgets/account_card.dart"
// AppColors.inaccessible.lightBlueOnWhite = #90CAF9
// Contrast ratio on white (#FFFFFF): ~2.0:1
// WCAG AA requires: 4.5:1
// Verdict: FAIL
Icon(
  _iconForType(account.type),
  color: inaccessible.lightBlueOnWhite, // #90CAF9 — ~2.0:1 on white
  size: 32,
),
// ...
Text(
  account.name,
  style: TextStyle(
    color: inaccessible.lightBlueOnWhite, // FAILS WCAG AA
    fontWeight: FontWeight.w600,
  ),
),
Text(
  '£${account.balance.toStringAsFixed(2)}',
  style: TextStyle(
    color: inaccessible.lightBlueOnWhite, // FAILS WCAG AA
    fontSize: 18,
  ),
),
```

  </TabItem>
  <TabItem value="after" label="After — passes WCAG 1.4.3">

```dart title="lib/screens/account_overview/widgets/account_card.dart"
// AppColors.primary = #1565C0 — contrast on white: ~7.3:1 ✓
// AppColors.textPrimary = #121212 — contrast on white: ~18.1:1 ✓
// Both pass WCAG AA (4.5:1) and WCAG AAA (7:1)
Icon(
  _iconForType(account.type),
  color: AppColors.primary, // #1565C0 — 7.3:1 on white ✓
  size: 32,
),
// ...
Text(
  account.name,
  style: const TextStyle(
    color: AppColors.textPrimary, // #121212 — 18.1:1 on white ✓
    fontWeight: FontWeight.w600,
  ),
),
Text(
  '£${account.balance.toStringAsFixed(2)}',
  style: const TextStyle(
    color: AppColors.textPrimary, // 18.1:1 on white ✓
    fontSize: 18,
  ),
),
```

  </TabItem>
</Tabs>

## Section 2: Text That Scales

Users in iOS Settings → Display & Brightness → Text Size, or Android Settings → Display → Font Size, can increase the system font size. Flutter respects this automatically via `MediaQuery.textScalerOf`. The problem arises when code explicitly overrides this.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

The old `textScaleFactor` parameter (now replaced by `textScaler` in Flutter 3.x) could be set to `1.0` to force the default size regardless of the user's preference. This is an accessibility violation under WCAG 1.4.4 ("Resize text") and should never be used.

Remove any explicit scale overrides and test at 200% font size — that's the WCAG 1.4.4 success criterion requirement.

  </div>
</div>

<Tabs>
  <TabItem value="before" label="Before — ignores user preference" default>

```dart title="lib/screens/account_overview/widgets/account_card.dart"
// Bad: forces the default scale regardless of the user's system settings.
// Users who rely on large text for visibility have their preference silently ignored.
Text(
  accountName,
  textScaleFactor: 1.0, // Overrides the user's accessibility settings
  style: const TextStyle(fontSize: 16),
)
```

  </TabItem>
  <TabItem value="after" label="After — respects user preference">

```dart title="lib/screens/account_overview/widgets/account_card.dart"
// Good: no explicit scale factor — Flutter applies MediaQuery automatically.
// If the user has set 150% font size, this text will be 150% of 16sp.
Text(
  account.name,
  style: const TextStyle(
    color: AppColors.textPrimary,
    fontWeight: FontWeight.w600,
    // fontSize: 16 — implied default, no scale override
  ),
)
```

  </TabItem>
</Tabs>

<div className="ab-callout ab-callout--amber">
  <div className="ab-callout__header">Why this matters</div>
  <p>Users increase their font size because they need it. Overriding that preference — silently, without any indication — is a significant accessibility barrier. WCAG 1.4.4 requires that text can be resized up to 200% without loss of content or functionality.</p>
</div>

## Section 3: Layouts That Don't Break at Large Text

Respecting text scale is only half the battle — your layout must also handle the extra space. Fixed-height containers that look fine at 100% scale will overflow at 200%.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

The patterns to avoid:
- `SizedBox(height: 48)` as a fixed container for text content
- `height:` constraints on `Row` or `Container` that hold text

The patterns to use instead:
- `ConstrainedBox(constraints: BoxConstraints(minHeight: 48))` — sets a minimum but allows growth
- `Flexible` or `Expanded` inside `Row`/`Column` — lets content size itself
- Let `Card` and `ListTile` size themselves based on content

</div>
</div>

<Tabs>
  <TabItem value="before" label="Before — overflows at large text" default>

```dart title="lib/screens/account_overview/widgets/account_card.dart"
// This looks fine at 100% but clips text at 200% font size.
// The SizedBox height is fixed — text can't grow beyond 56px.
SizedBox(
  height: 56,
  child: Row(
    children: [
      const Icon(Icons.account_balance),
      Text(accountName),
      const Spacer(),
      Text(balanceText),
    ],
  ),
)
```

  </TabItem>
  <TabItem value="after" label="After — grows with text">

```dart title="lib/screens/account_overview/widgets/account_card.dart"
// minHeight: 56 sets the default size but allows the container to grow.
// Expanded(child: Text(...)) wraps long names instead of clipping them.
ConstrainedBox(
  constraints: const BoxConstraints(minHeight: 56),
  child: Row(
    crossAxisAlignment: CrossAxisAlignment.center,
    children: [
      const Icon(Icons.account_balance),
      const SizedBox(width: 12),
      Expanded(
        child: Text(
          accountName,
          style: const TextStyle(color: AppColors.textPrimary),
        ),
      ),
      Text(
        balanceText,
        style: const TextStyle(color: AppColors.textPrimary),
      ),
    ],
  ),
)
```

  </TabItem>
</Tabs>
