# Forms That Work for Everyone

> *"Forms are where accessibility really saves people."*

**Estimated time:** ~25 minutes | **Focus:** Login + Transfer | **Branch:** `chapter-5-forms`

---

Forms are where accessibility failures hit hardest. An inaccessible login screen locks users out of their own account. An inaccessible transfer form can cost them real money. By the end of this chapter, AccessBank's Login and Transfer forms will properly label every field, announce errors through the screen reader, move focus to problems, and use smart keyboard types to make data entry as effortless as possible.

---

## Section 1: Label Everything

The most common form accessibility mistake is relying on `hintText` alone. Hint text disappears the moment a user starts typing — leaving them with no label to refer to while they edit.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

Use `InputDecoration.labelText` for the persistent floating label. This gives you two things at once: a visible label that stays on screen, and a semantic label that the screen reader announces as *"Email, text field, double tap to edit"* instead of a generic *"text field"*.

  </div>
</div>

<Tabs>
  <TabItem value="before" label="Before — inaccessible">

```dart title="lib/screens/login/widgets/login_form.dart"
// Hint text vanishes on focus; screen reader says "text field"
TextField(
  controller: _emailController,
  decoration: const InputDecoration(
    hintText: 'Email',
    border: OutlineInputBorder(),
  ),
  keyboardType: TextInputType.emailAddress,
),
```

  </TabItem>
  <TabItem value="after" label="After — accessible" default>

```dart title="lib/screens/login/widgets/login_form.dart"
// Label floats above the field and persists; screen reader says "Email, text field"
TextField(
  controller: _emailController,
  focusNode: _emailFocusNode,
  decoration: const InputDecoration(
    labelText: 'Email',
    hintText: 'you@example.com',
    border: OutlineInputBorder(),
  ),
  keyboardType: TextInputType.emailAddress,
  autofillHints: const [AutofillHints.email],
  textInputAction: TextInputAction.next,
  onSubmitted: (_) => _passwordFocusNode.requestFocus(),
),
```

  </TabItem>
</Tabs>

<div className="ab-callout ab-callout--amber">
  <div className="ab-callout__header">Why this matters</div>
  <p>VoiceOver and TalkBack users navigate fields by swiping. If a field only has a hint text, the user hears "text field" — no context at all. <code>labelText</code> gives the screen reader a persistent, meaningful label even when the field is filled in.</p>
</div>

## Section 2: Announce Errors

When validation fails, sighted users see red error text appear below the field. Screen reader users hear nothing — the form just silently refuses to submit.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

`SemanticsService.sendAnnouncement()` inserts a spoken message into the platform's accessibility announcement queue — like a live-region update that is spoken without requiring the user to move focus. Call it immediately after setting error state so the user hears what went wrong.

  </div>
</div>

<div className="ab-step">
  <div className="ab-step__badge">2</div>
  <div className="ab-step__content">

For a persistent error widget that auto-announces when it appears, wrap it in `Semantics(liveRegion: true)`. This is the belt-and-suspenders approach — the announcement fires both from `sendAnnouncement` and from the live region when the widget rebuilds.

  </div>
</div>

<Tabs>
  <TabItem value="before" label="Before — inaccessible">

```dart title="lib/screens/login/widgets/login_form.dart"
void _submitInaccessible() {
  // Silent validation — no announcement to screen readers
  if (_emailController.text.isEmpty || _passwordController.text.isEmpty) {
    setState(() => _errorText = 'Invalid credentials');
    return;
  }
  widget.onSubmit();
}

// Inaccessible: plain Text — no semantics announcement, no live region
Text(
  _errorText!,
  style: const TextStyle(color: Colors.red, fontSize: 12),
),
```

  </TabItem>
  <TabItem value="after" label="After — accessible" default>

```dart title="lib/screens/login/widgets/login_form.dart"
void _submitAccessible() {
  if (_emailController.text.isEmpty || _passwordController.text.isEmpty) {
    const message = 'Sign in failed. Please enter your email and password.';
    setState(() => _errorText = message);
    // Announce the error to screen readers immediately
    SemanticsService.sendAnnouncement(
      View.of(context),
      message,
      TextDirection.ltr,
    );
    // Move focus to the first empty field
    if (_emailController.text.isEmpty) {
      _emailFocusNode.requestFocus();
    } else {
      _passwordFocusNode.requestFocus();
    }
    return;
  }
  widget.onSubmit();
}

// Accessible: liveRegion ensures announcement fires on rebuild too
Semantics(
  liveRegion: true,
  child: Text(
    _errorText!,
    style: TextStyle(
      color: Theme.of(context).colorScheme.error,
      fontSize: 13,
    ),
  ),
),
```

  </TabItem>
</Tabs>

<div className="ab-callout ab-callout--blue">
  <div className="ab-callout__header">💡 Try it yourself</div>
  <p>Enable TalkBack (Android) or VoiceOver (iOS) and try to submit the login form with empty fields. In the "before" version you'll hear nothing. In the "after" version the error is spoken immediately, and focus jumps to the empty email field.</p>
</div>

## Section 3: Focus to the Error

Announcing an error is a good start, but the user still needs to find the problematic field. Moving focus automatically to the first field that has an error puts the user in the right place to fix it.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

Attach a `FocusNode` to each field. After validation, call `requestFocus()` on the node for the first failing field. The screen reader will announce the field name and move the user's attention there in one step.

  </div>
</div>

<Tabs>
  <TabItem value="before" label="Before — user hunts for the error">

```dart title="lib/screens/login/widgets/login_form.dart"
// No FocusNodes — user must swipe back through the form to find the error
TextField(
  controller: _emailController,
  decoration: const InputDecoration(hintText: 'Email'),
),
TextField(
  controller: _passwordController,
  decoration: const InputDecoration(hintText: 'Password'),
  obscureText: true,
),
```

  </TabItem>
  <TabItem value="after" label="After — focus jumps to the error" default>

```dart title="lib/screens/login/widgets/login_form.dart"
// FocusNodes enable programmatic focus management
final _emailFocusNode = FocusNode();
final _passwordFocusNode = FocusNode();

TextField(
  controller: _emailController,
  focusNode: _emailFocusNode,
  decoration: const InputDecoration(labelText: 'Email'),
  textInputAction: TextInputAction.next,
  onSubmitted: (_) => _passwordFocusNode.requestFocus(),
),
TextField(
  controller: _passwordController,
  focusNode: _passwordFocusNode,
  decoration: const InputDecoration(labelText: 'Password'),
  obscureText: true,
  textInputAction: TextInputAction.done,
  onSubmitted: (_) => _submit(),
),
```

  </TabItem>
</Tabs>

<div className="ab-callout ab-callout--green">
  <div className="ab-callout__header">✓ Checkpoint</div>
  <p>With TalkBack or VoiceOver active, submit the login form with both fields empty. You should hear the error announcement immediately, and focus should land on the Email field ready for input — without any swiping required.</p>
</div>

## Section 4: Live Validation on the Transfer Form

The Transfer screen's amount field accepts a dollar value but says nothing until the user hits Confirm. Real-time validation with an accessible announcement gives users immediate feedback as they type.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

Use `onChanged` to validate and `SemanticsService.sendAnnouncement()` to announce the result. The Transfer form's accessible step already demonstrates this — it validates that the amount is a positive number and fires an announcement if not.

  </div>
</div>

<Tabs>
  <TabItem value="before" label="Before — no live feedback">

```dart title="lib/screens/transfer/widgets/transfer_form.dart"
// Placeholder-only, number keyboard, no live validation
TextField(
  decoration: const InputDecoration(
    hintText: 'Amount',
    border: OutlineInputBorder(),
  ),
  keyboardType: TextInputType.number,
  onChanged: onAmountChanged,
),
```

  </TabItem>
  <TabItem value="after" label="After — live validation + announcement" default>

```dart title="lib/screens/transfer/widgets/transfer_form.dart"
String? _amountError;

void _validateAmount(String value) {
  widget.onAmountChanged(value);
  final parsed = double.tryParse(value);
  if (value.isNotEmpty && (parsed == null || parsed <= 0)) {
    const msg = 'Please enter a valid amount greater than zero';
    setState(() => _amountError = msg);
    SemanticsService.sendAnnouncement(
      View.of(context),
      msg,
      TextDirection.ltr,
    );
  } else {
    setState(() => _amountError = null);
  }
}

TextField(
  decoration: InputDecoration(
    labelText: 'Amount',
    hintText: '0.00',
    prefixText: '\$',
    border: const OutlineInputBorder(),
    errorText: _amountError,   // Shows red error text below field
  ),
  keyboardType: const TextInputType.numberWithOptions(decimal: true),
  textInputAction: TextInputAction.next,
  onChanged: _validateAmount,
),
```

  </TabItem>
</Tabs>

<div className="ab-callout ab-callout--amber">
  <div className="ab-callout__header">Why this matters</div>
  <p>Live validation catches errors at the point of entry rather than at submission. For screen reader users, this means one announcement at the right moment rather than a wall of errors to navigate after tapping Confirm. For users with motor impairments, it means fewer form submissions and less re-entry.</p>
</div>

## Section 5: Smart Input Types and Autofill

Setting the right `keyboardType` and `autofillHints` on each field improves the experience for everyone — especially users with motor impairments who find typing difficult.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

`autofillHints` enables iOS AutoFill and Android Autofill Service to populate credentials from password managers. Combined with a logical `textInputAction` chain (`next` → `next` → `done`), the user can move through the entire form using only the keyboard return key.

  </div>
</div>

<Tabs>
  <TabItem value="before" label="Before — generic inputs">

```dart title="lib/screens/login/widgets/login_form.dart"
// Generic keyboard; no autofill; no logical next-field action
TextField(
  decoration: const InputDecoration(hintText: 'Email'),
  controller: _emailController,
),
TextField(
  decoration: const InputDecoration(hintText: 'Password'),
  controller: _passwordController,
  obscureText: true,
),
// Vague button label — no context for screen readers
FilledButton(onPressed: _submit, child: const Text('Go')),
```

  </TabItem>
  <TabItem value="after" label="After — smart inputs" default>

```dart title="lib/screens/login/widgets/login_form.dart"
// Email keyboard + autofill + logical next action
TextField(
  controller: _emailController,
  focusNode: _emailFocusNode,
  decoration: const InputDecoration(
    labelText: 'Email',
    hintText: 'you@example.com',
    border: OutlineInputBorder(),
  ),
  keyboardType: TextInputType.emailAddress,
  autofillHints: const [AutofillHints.email],
  textInputAction: TextInputAction.next,
  onSubmitted: (_) => _passwordFocusNode.requestFocus(),
),
// Password keyboard + autofill + submit on done
TextField(
  controller: _passwordController,
  focusNode: _passwordFocusNode,
  decoration: const InputDecoration(
    labelText: 'Password',
    border: OutlineInputBorder(),
  ),
  obscureText: true,
  autofillHints: const [AutofillHints.password],
  textInputAction: TextInputAction.done,
  onSubmitted: (_) => _submit(),
),
// Descriptive button label — clear context for screen readers
SizedBox(
  height: 48, // Minimum 48×48 touch target
  child: FilledButton(onPressed: _submit, child: const Text('Sign In')),
),
```

  </TabItem>
</Tabs>

<div className="ab-callout ab-callout--blue">
  <div className="ab-callout__header">💡 Try it yourself</div>
  <p>On a real iOS or Android device, open the "after" version of the Login screen and tap the email field. Does the email keyboard appear? Does tapping "next" on the keyboard move you to the password field? Does iOS offer to AutoFill credentials? These should all work in the accessible version.</p>
</div>

## Check Your Understanding

<details>
<summary>Question 1: Why should you use <code>InputDecoration.labelText</code> instead of only <code>hintText</code> for form fields?</summary>

**A) labelText has better typography**
**B) hintText is not supported on all platforms**
**C) labelText stays visible as a floating label when the user types; hintText disappears** ✅
**D) labelText is required by Flutter**

`hintText` disappears as soon as the user starts typing, leaving them with no label to refer back to. `labelText` floats above the field and remains visible, which benefits both sighted users and screen reader users who need context while editing.

</details>

<details>
<summary>Question 2: What does <code>SemanticsService.sendAnnouncement()</code> do?</summary>

**A) Adds a tooltip to the nearest Semantics widget**
**B) Inserts a spoken message into the accessibility announcement queue without requiring focus change** ✅
**C) Logs an accessibility event to the console**
**D) Sets the label of the currently focused widget**

`SemanticsService.sendAnnouncement()` sends a live-region announcement to the platform's accessibility service. The message is spoken by the screen reader without the user having to move focus — ideal for error messages and status updates.

</details>

<details>
<summary>Question 3: Which combination of improvements makes the Transfer amount field most accessible?</summary>

**A) labelText + decimal number keyboard + live error announcement** ✅
**B) hintText only**
**C) Large font size + border colour change**
**D) Placeholder text + onSubmitted callback**

The accessible amount field combines: a visible `labelText`, a `numberWithOptions(decimal: true)` keyboard, and a `SemanticsService.sendAnnouncement()` call in `onChanged` for live error feedback — giving sighted, screen reader, and motor-impaired users the best experience.

</details>
