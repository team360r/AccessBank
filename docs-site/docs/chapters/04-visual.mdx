# See Clearly

> *"My grandma could actually use this now."*

**Estimated time:** ~20 minutes | **Focus:** Account Overview + Theme | **Branch:** `chapter-4-visual`

---

Visual accessibility goes far beyond screen readers. In this chapter you'll tackle the problems that affect users who *can* see the screen — just not as clearly as you might assume. That includes older adults, people with low vision, users in bright sunlight, and anyone who's increased their device font size.

There are more of these users than you might think. One in three people over 65 have some form of vision impairment. And everyone — regardless of vision — benefits from text they can actually read.

---

## Section 1: Contrast Matters

The original AccessBank colour scheme uses `AppColors.inaccessible.lightBlueOnWhite` (`#90CAF9`) for account names and balances on white cards. That's a contrast ratio of roughly **2.0:1** against white. WCAG 2.1 requires **4.5:1** for normal-sized body text.

That's not a near-miss. It's less than half the required contrast.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

WCAG 2.1 Success Criterion 1.4.3 sets these minimum contrast ratios:

- **4.5:1** for normal text (below 18pt regular or 14pt bold)
- **3:1** for large text (18pt regular or 14pt bold and above)
- **3:1** for user interface components and graphical objects

These aren't arbitrary numbers. They represent the threshold at which most people with moderate vision loss — including many older adults — can still read text comfortably.

  </div>
</div>

<div className="ab-step">
  <div className="ab-step__badge">2</div>
  <div className="ab-step__content">

The fix in AccessBank is to switch from the inaccessible palette to `AppColors.primary` and `AppColors.textPrimary` — deep blue and near-black respectively, both with contrast ratios well above 4.5:1 against the white card background.

  </div>
</div>

<Tabs>
  <TabItem value="before" label="Before — fails WCAG 1.4.3">

```dart title="lib/screens/account_overview/widgets/account_card.dart"
// AppColors.inaccessible.lightBlueOnWhite = #90CAF9
// Contrast ratio on white (#FFFFFF): ~2.0:1
// WCAG AA requires: 4.5:1
// Verdict: FAIL
Icon(
  _iconForType(account.type),
  color: inaccessible.lightBlueOnWhite, // #90CAF9 — ~2.0:1 on white
  size: 32,
),
// ...
Text(
  account.name,
  style: TextStyle(
    color: inaccessible.lightBlueOnWhite, // FAILS WCAG AA
    fontWeight: FontWeight.w600,
  ),
),
Text(
  '\$${account.balance.toStringAsFixed(2)}',
  style: TextStyle(
    color: inaccessible.lightBlueOnWhite, // FAILS WCAG AA
    fontSize: 18,
  ),
),
```

  </TabItem>
  <TabItem value="after" label="After — passes WCAG 1.4.3" default>

```dart title="lib/screens/account_overview/widgets/account_card.dart"
// AppColors.primary = #1565C0 — contrast on white: ~7.3:1 ✓
// AppColors.textPrimary = #121212 — contrast on white: ~18.1:1 ✓
// Both pass WCAG AA (4.5:1) and WCAG AAA (7:1)
Icon(
  _iconForType(account.type),
  color: AppColors.primary, // #1565C0 — 7.3:1 on white ✓
  size: 32,
),
// ...
Text(
  account.name,
  style: const TextStyle(
    color: AppColors.textPrimary, // #121212 — 18.1:1 on white ✓
    fontWeight: FontWeight.w600,
  ),
),
Text(
  '\$${account.balance.toStringAsFixed(2)}',
  style: const TextStyle(
    color: AppColors.textPrimary, // 18.1:1 on white ✓
    fontSize: 18,
  ),
),
```

  </TabItem>
</Tabs>

<div className="ab-callout ab-callout--amber">
  <div className="ab-callout__header">Why this matters</div>
  <p>4.5:1 is not just a compliance checkbox — it's the threshold where most people with moderate vision loss can still read text comfortably. One in three people over 65 have some form of vision impairment. The "stylish" light-blue palette we removed was literally unreadable for a large portion of AccessBank's potential users.</p>
</div>

## Section 2: Text That Scales

Users in iOS Settings → Display & Brightness → Text Size, or Android Settings → Display → Font Size, can increase the system font size. Flutter respects this automatically via `MediaQuery.textScalerOf`. The problem arises when code explicitly overrides this.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

The old `textScaleFactor` parameter (now replaced by `textScaler` in Flutter 3.x) could be set to `1.0` to force the default size regardless of the user's preference. This is an accessibility violation under WCAG 1.4.4 ("Resize text") and should never be used.

Remove any explicit scale overrides and test at 200% font size — that's the WCAG 1.4.4 success criterion requirement.

  </div>
</div>

<Tabs>
  <TabItem value="before" label="Before — ignores user preference">

```dart title="lib/screens/account_overview/widgets/account_card.dart"
// Bad: forces the default scale regardless of the user's system settings.
// Users who rely on large text for visibility have their preference silently ignored.
Text(
  accountName,
  textScaleFactor: 1.0, // Overrides the user's accessibility settings
  style: const TextStyle(fontSize: 16),
)
```

  </TabItem>
  <TabItem value="after" label="After — respects user preference" default>

```dart title="lib/screens/account_overview/widgets/account_card.dart"
// Good: no explicit scale factor — Flutter applies MediaQuery automatically.
// If the user has set 150% font size, this text will be 150% of 16sp.
Text(
  account.name,
  style: const TextStyle(
    color: AppColors.textPrimary,
    fontWeight: FontWeight.w600,
    // fontSize: 16 — implied default, no scale override
  ),
)
```

  </TabItem>
</Tabs>

<div className="ab-callout ab-callout--amber">
  <div className="ab-callout__header">Why this matters</div>
  <p>Users increase their font size because they need it. Overriding that preference — silently, without any indication — is a significant accessibility barrier. WCAG 1.4.4 requires that text can be resized up to 200% without loss of content or functionality.</p>
</div>

## Section 3: Layouts That Don't Break at Large Text

Respecting text scale is only half the battle — your layout must also handle the extra space. Fixed-height containers that look fine at 100% scale will overflow at 200%.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

The patterns to avoid:
- `SizedBox(height: 48)` as a fixed container for text content
- `height:` constraints on `Row` or `Container` that hold text

The patterns to use instead:
- `ConstrainedBox(constraints: BoxConstraints(minHeight: 48))` — sets a minimum but allows growth
- `Flexible` or `Expanded` inside `Row`/`Column` — lets content size itself
- Let `Card` and `ListTile` size themselves based on content

</div>
</div>

<Tabs>
  <TabItem value="before" label="Before — overflows at large text">

```dart title="lib/screens/account_overview/widgets/account_card.dart"
// This looks fine at 100% but clips text at 200% font size.
// The SizedBox height is fixed — text can't grow beyond 56px.
SizedBox(
  height: 56,
  child: Row(
    children: [
      const Icon(Icons.account_balance),
      Text(accountName),
      const Spacer(),
      Text(balanceText),
    ],
  ),
)
```

  </TabItem>
  <TabItem value="after" label="After — grows with text" default>

```dart title="lib/screens/account_overview/widgets/account_card.dart"
// minHeight: 56 sets the default size but allows the container to grow.
// Expanded(child: Text(...)) wraps long names instead of clipping them.
ConstrainedBox(
  constraints: const BoxConstraints(minHeight: 56),
  child: Row(
    crossAxisAlignment: CrossAxisAlignment.center,
    children: [
      const Icon(Icons.account_balance),
      const SizedBox(width: 12),
      Expanded(
        child: Text(
          accountName,
          style: const TextStyle(color: AppColors.textPrimary),
        ),
      ),
      Text(
        balanceText,
        style: const TextStyle(color: AppColors.textPrimary),
      ),
    ],
  ),
)
```

  </TabItem>
</Tabs>

<div className="ab-callout ab-callout--blue">
  <div className="ab-callout__header">💡 Try it yourself</div>
  <p>Go to your device's Settings → Display → Font Size (Android) or Settings → Display &amp; Brightness → Text Size (iOS) and set it to the maximum. Switch back to AccessBank and look at the Account Overview. Does anything clip or overflow? That's your to-do list.</p>
</div>

## Section 4: Dark Mode Done Right

Dark mode isn't just a visual preference — for many users with photosensitivity or migraine conditions, it's a necessity. But dark mode introduces its own contrast challenges. Colours that pass 4.5:1 on white backgrounds often fail on dark surfaces.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

Each theme variant needs its own contrast-tested palette. The `AppColors` class in AccessBank defines accessible colours for the light theme — but a dark theme needs different values for `onSurface`, `textSecondary`, and other roles.

  </div>
</div>

<Tabs>
  <TabItem value="before" label="Before — fails in dark mode">

```dart title="lib/theme/app_theme.dart"
// onSurface: #9E9E9E on dark surface #1E1E1E
// Contrast ratio: ~3.8:1 — FAILS WCAG AA for normal text (needs 4.5:1)
ThemeData darkTheme = ThemeData.dark().copyWith(
  colorScheme: const ColorScheme.dark(
    surface: Color(0xFF1E1E1E),
    onSurface: Color(0xFF9E9E9E), // ~3.8:1 — fails WCAG AA
  ),
);
```

  </TabItem>
  <TabItem value="after" label="After — passes in dark mode" default>

```dart title="lib/theme/app_theme.dart"
// onSurface: #D4D4D4 on dark surface #1E1E1E
// Contrast ratio: ~9.2:1 — passes WCAG AA AND AAA
ThemeData darkTheme = ThemeData.dark().copyWith(
  colorScheme: const ColorScheme.dark(
    surface: Color(0xFF1E1E1E),
    onSurface: Color(0xFFD4D4D4), // ~9.2:1 — passes WCAG AAA ✓
  ),
);
```

  </TabItem>
</Tabs>

<div className="ab-callout ab-callout--blue">
  <div className="ab-callout__header">💡 Material 3 shortcut</div>
  <p>If you use Material 3's dynamic colour system with a seed colour (<code>ColorScheme.fromSeed(seedColor: AppColors.primary)</code>), Flutter generates a full colour scheme with accessible contrast ratios built in — for both light and dark modes. It's the easiest way to get contrast right without manually testing every colour pair.</p>
</div>

## Section 5: Beyond Colour — Don't Rely on Colour Alone

About 300 million people are colour-blind — most commonly with red-green confusion (deuteranopia or protanopia). If your app uses colour as the *only* way to communicate information, those users miss it entirely.

<div className="ab-step">
  <div className="ab-step__badge">1</div>
  <div className="ab-step__content">

WCAG 1.4.1 ("Use of Colour") states that colour must not be the only visual means of conveying information, indicating an action, prompting a response, or distinguishing a visual element.

The fix: always pair colour with a secondary indicator — an icon, a text label, a shape, or a pattern.

  </div>
</div>

<div className="ab-step">
  <div className="ab-step__badge">2</div>
  <div className="ab-step__content">

In AccessBank, transaction amounts show green for positive and red for negative. A user with red-green colour-blindness sees both as the same brownish-grey. Adding directional icons (`arrow_upward` / `arrow_downward`) makes the distinction clear regardless of colour perception.

  </div>
</div>

<Tabs>
  <TabItem value="before" label="Before — colour only">

```dart title="lib/widgets/balance_indicator.dart"
// Colour alone — invisible to users with red-green colour blindness.
// Both "positive" and "negative" indicators look identical to ~8% of men.
Container(
  width: 8,
  height: 8,
  decoration: BoxDecoration(
    color: isPositive ? Colors.green : Colors.red,
    shape: BoxShape.circle,
  ),
)
```

  </TabItem>
  <TabItem value="after" label="After — colour + icon + semantic label" default>

```dart title="lib/widgets/balance_indicator.dart"
// Colour + directional icon + semantic label.
// The meaning is clear regardless of how you perceive colour.
Row(
  mainAxisSize: MainAxisSize.min,
  children: [
    Icon(
      isPositive ? Icons.arrow_upward : Icons.arrow_downward,
      color: isPositive ? AppColors.success : AppColors.error,
      size: 14,
      semanticLabel: isPositive ? 'Positive balance' : 'Overdrawn',
    ),
    const SizedBox(width: 4),
    Container(
      width: 8,
      height: 8,
      decoration: BoxDecoration(
        // Use accessible colour values with sufficient contrast
        color: isPositive ? AppColors.success : AppColors.error,
        shape: BoxShape.circle,
      ),
    ),
  ],
)
```

  </TabItem>
</Tabs>

<div className="ab-callout ab-callout--amber">
  <div className="ab-callout__header">Why this matters</div>
  <p>About 300 million people are colour-blind globally — roughly 8% of men and 0.5% of women. In a banking app, distinguishing positive from negative balances, successful from failed transactions, and healthy from overdrawn accounts is critical. Colour alone never communicates meaning reliably.</p>
</div>

<div className="ab-callout ab-callout--blue">
  <div className="ab-callout__header">💡 Try it yourself</div>
  <p>There are many colour-blindness simulation tools you can use to preview your app. <a href="https://www.toptal.com/designers/colorfilter">Toptal Color Blind Filter</a> can simulate different types of colour blindness on screenshots. Take a screenshot of AccessBank and run it through the red-green blind simulation. Can you still distinguish positive from negative amounts?</p>
</div>

<div className="ab-callout ab-callout--green">
  <div className="ab-callout__header">✓ Checkpoint</div>
  <p>After completing this chapter: (1) all text in the app passes a 4.5:1 contrast check (use a tool to verify), (2) no <code>textScaleFactor: 1.0</code> overrides exist in the codebase, (3) account cards use <code>ConstrainedBox</code> rather than fixed-height <code>SizedBox</code>, and (4) balance indicators use both colour and icon to communicate direction. Test at 200% font size to confirm nothing overflows.</p>
</div>

## Check Your Understanding

<details>
<summary>Question 1: What is the WCAG AA minimum contrast ratio for normal-sized body text?</summary>

**A) 3:1**
**B) 4.5:1** ✅
**C) 7:1**
**D) 2:1**

WCAG 2.1 Success Criterion 1.4.3 requires a 4.5:1 contrast ratio for normal text (below 18pt regular or 14pt bold). Large text (18pt regular or 14pt bold and above) requires a minimum of 3:1. WCAG AAA for normal text requires 7:1.

</details>

<details>
<summary>Question 2: What happens when you set textScaleFactor: 1.0 on a Text widget?</summary>

**A) The text becomes slightly larger**
**B) The user's system font-size preference is ignored** ✅
**C) The text is hidden from screen readers**
**D) The text is rendered at the system default size**

Setting `textScaleFactor: 1.0` (or the newer `textScaler: TextScaler.noScaling`) overrides the device's system font-size setting, forcing text to render at the default scale even when the user has requested larger text. This is an accessibility violation under WCAG 1.4.4.

</details>

**D) 2.4.6 Headings and Labels**
